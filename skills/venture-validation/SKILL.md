# Venture Validation

## Overview

Use this skill to turn vague startup discussion into a validation-driven judgment.
The goal is not to admire an idea. The goal is to identify the assumptions that could kill it, decide what must be tested first, and produce a clear go, pivot, or kill recommendation.

## Core Principle

Do not analyze every question equally.
Prioritize the assumptions that, if false, would make the rest of the business irrelevant.

Default priority:

1. Demand risk
2. Solution-value risk
3. Business model risk
4. Growth risk
5. Defensibility risk

This default order changes only when the project type changes materially.

## Workflow

### 1. Set the business target

First identify what kind of project this is:

- small cashflow business
- growth-first venture-style business
- unclear or mixed goal

Also identify:

- expected scale
- time horizon
- available money
- team capability
- what success means

Do not evaluate a side business with venture standards.
Do not evaluate a venture-style idea with side-business standards.

### 2. Map the business in five modules

Always structure the idea into:

1. demand
2. solution
3. business model
4. growth
5. defensibility

Treat these five modules as five different business questions:

- demand = origin problem: who, in what situation, suffers from what problem
- solution = value problem: why this offer is meaningfully better
- business model = efficiency problem: whether the business can make money and keep making money
- growth = replication problem: whether customer acquisition can repeat and scale
- defensibility = endgame problem: why the model can keep winning under competition

For each module, ask:

- What is being assumed?
- What evidence already exists?
- What remains uncertain?
- If this fails, how much of the business collapses?

### 3. Find the highest-risk assumptions

For each important assumption, rate it using these five lenses:

- uncertainty: how weak is the evidence?
- impact scope: does failure break one feature or the whole model?
- validation cost: how expensive is it to learn the truth?
- failure probability: how likely is the assumption to be wrong?
- irreversibility: how painful is it if the assumption fails late?

Then sort assumptions by:

1. fatality
2. validation urgency

Use this rule of thumb:

- demand risk usually outranks model risk
- solution-value risk usually outranks growth risk
- front-of-chain assumptions usually outrank later optimization assumptions

Output only 1-3 highest-risk assumptions, not a long list.

### 4. Design low-cost validation

For each highest-risk assumption, design the cheapest credible test.

Allowed test types:

- user interviews
- landing page test
- waitlist or lead collection
- prototype or MVP trial
- pricing test
- pre-order or deposit
- small paid acquisition test
- manual concierge delivery

Each test should specify:

- what assumption is being tested
- what evidence would support it
- what evidence would reject it
- what threshold counts as good enough
- what the next decision will be

### 5. Decide go, pivot, or kill

Use evidence, not energy, to decide.

Default decision logic:

- **Go**: core demand and solution assumptions are strongly supported, and there is at least an early path to viable unit economics and repeatable acquisition
- **Pivot**: some valuable part is real, but the current user, problem framing, offer, model, or acquisition path is wrong
- **Kill**: the core problem is weak, the value proposition is rejected, the model has no plausible path to health, or the team lacks non-negotiable capabilities

- [references/decision-thresholds.md](references/decision-thresholds.md)

## Project Type Adjustment

For **small cashflow businesses**:

- prioritize profitability and cashflow
- require healthier unit economics earlier
- accept narrower markets
- reduce obsession with giant TAM
- judge defensibility more through relationship depth and operational strength

For **high-growth venture-style businesses**:

- prioritize market size and growth engine
- allow delayed profitability if the path is credible
- weigh defensibility and structural advantage more heavily
- care much more about scale dynamics and market capture

Use the semantic differences deliberately:

- small cashflow businesses care more about precise demand, profitable delivery, and stable repeat acquisition
- venture-style businesses care more about large demand, nonlinear growth, and structural defensibility

## Output Format

Default output sections:

1. One-line judgment
2. Project type
3. Five-module breakdown
4. Highest-risk assumptions
5. Validation plan
6. Go / Pivot / Kill decision
7. Immediate next move

If the user wants a more structured scorecard, use:
- [references/scoring-sheet.md](references/scoring-sheet.md)

## Chinese-First Usage

If the operator prefers Chinese, keep the exact same internal logic and section order, but write the report in Chinese.
If the user provides very little information, ask only the minimum key questions needed to classify project type and identify the first validation target.

## Common Mistakes

- Treating user interest as proof of painful demand
- Treating compliments as proof of willingness to pay
- Treating traffic as proof of business quality
- Optimizing growth before proving value
- Using one generic framework for both small cashflow and venture-style projects
- Listing many risks instead of isolating the few that matter most

## Fast Reminder

The right question is not:
"Is this idea good?"

The right question is:
"Which assumptions must be true for this to work, and which one should be tested first?"
